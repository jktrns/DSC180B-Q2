Training-based versus training-free
differential privacy for data synthesis

Mehak Kapur, Hana Tjendrawasi, Jason Tran, Phuc Tran
{mekapur,htjendrawasi,jat037,pct001}@ucsd.edu
Halıcıoğlu Data Science Institute, UC San Diego

Code Report (PDF) Poster

Differentially private synthetic data generation promises to resolve the tension between data utility and individual privacy, enabling the release of datasets that preserve the statistical properties analysts need while bounding what any adversary can learn about a single record. Two paradigms have emerged: training-based methods, which inject calibrated noise during model optimization, and training-free methods, which leverage foundation models through black-box API access. We investigate both approaches on Intel’s Driver and Client Applications (DCA) telemetry corpus, evaluating against a benchmark of 21 analytical SQL queries representative of production business intelligence workloads.

Introduction

Why does this matter?

Device telemetry—hardware specs, usage patterns, network activity—helps product teams improve software, but it contains sensitive information about individuals. Releasing raw telemetry poses privacy risks. Differential privacy (DP) lets us publish synthetic data that preserves statistical utility while mathematically bounding the information leaked about any single record.

The core question: which DP synthesis approach best preserves the utility of production SQL workloads on multi-table relational data?

The privacy-utility tension

Differential privacy adds carefully calibrated noise so that an algorithm’s output is nearly identical whether or not any single record is in the dataset. The privacy “cost” is measured by a parameter ε (epsilon)—smaller ε means stronger privacy but potentially lower utility. We use (ε, δ)-differential privacy with ε = 4.0 and δ = 10−5, matched across all methods for fair comparison.

Formal definition (click to expand)

A randomized mechanism M satisfies (ε, δ)-DP if for all neighboring datasets D, D′ (differing in one record) and all measurable sets S:

Pr[M(D) ∈ S] ≤ eε · Pr[M(D′) ∈ S] + δ

In practice, DP-SGD tracks privacy loss via Rényi Differential Privacy (RDP) accountants and converts to (ε, δ) at the end of training.

Two paradigms for synthetic data

  • Training-based methods inject calibrated noise during model optimization (e.g., DP-SGD with VAEs).
  • Training-free methods leverage foundation models through black-box API access (e.g., Private Evolution with LLMs).

We compare four concrete instantiations of these paradigms on a realistic multi-table telemetry benchmark.

Data handling

  • No data leakage: synthetic data is generated from the full dataset; benchmark queries test statistical fidelity, not prediction.
  • No private data in outputs: all published results are aggregate statistics or synthetic data. No individual records are exposed.
  • Preprocessing: continuous columns normalized to [0, 1]; categorical columns label-encoded. Missing values imputed per table.

Methods

1. Wide-table DP-VAE

Flatten all tables into a single wide table (left-join on system_id), then train a Variational Autoencoder (VAE) with DP-SGD. The VAE learns a compressed latent representation and generates synthetic records that preserve cross-table structure.

  • Strength: preserves foreign-key relationships by design.
  • Weakness: extreme sparsity (most columns are null after flattening) causes zero-inflation and mode collapse.

2. Per-table DP-SGD

Train a separate VAE for each table independently using DP-SGD. This avoids the sparsity problem of wide-table flattening but breaks cross-table relationships.

  • Strength: high within-table fidelity; each model is simpler and faster to train.
  • Weakness: foreign-key mismatches—a synthetic CPU record may reference a non-existent system.

3. MST (Maximum Spanning Tree)

A histogram-based baseline: estimate low-dimensional marginals under DP, build a dependency graph (Chow-Liu tree), and sample synthetic records that are consistent with the noisy marginals.

  • Strength: interpretable, fast, works well for categorical data with clear dependencies.
  • Weakness: exponential in table width; struggles with many continuous or high-cardinality columns.

4. Private Evolution

A training-free approach: use a large language model (LLM) to generate candidate synthetic records, then use the exponential mechanism (a DP algorithm) to select candidates that best match real data distributions.

  • Strength: no training required; can target specific query workloads.
  • Weakness: limited by the LLM’s training data—categories or values unseen by the model cannot be synthesized.

Privacy budget comparison

All methods use ε = 4.0, δ = 10−5. Training-based methods (DP-SGD) track cumulative privacy loss via Rényi DP accountants. Training-free methods (PE) spend their budget through the exponential mechanism’s selection step.

What the team built vs. reused

  • Built: the 21-query benchmark, the per-table DP-SGD pipeline, the Private Evolution integration, and the evaluation harness.
  • Reused: the MST implementation from the private-pgm library, the VAE architecture adapted from SDV, and Opacus for DP-SGD gradient clipping.

Key insights

What worked

  • Simple distributions: histogram queries (browser usage, RAM utilization) passed for MST and per-table DP-SGD. Total Variation distances were low (< 0.15).
  • Single-table aggregates: queries on independent features within one table were well-preserved by per-table DP-SGD.
  • MST on structured data: marginal-based synthesis excelled at queries with high-cardinality group-by keys.

What failed

  • Join-heavy aggregates: queries requiring cross-table joins failed across all methods due to foreign-key mismatches.
  • Wide-table sparsity: DP-VAE on the wide table suffered from zero-inflation—continuous distributions collapsed to near-zero values.
  • LLM knowledge gaps (PE): Private Evolution could not synthesize categories unseen in the LLM’s training data.

Interpretation guide: differences in pass rates of 2+ queries are significant. Score differences < 0.05 are within noise. The gap between Per-table/MST (6/21) and Wide/PE (1–2/8) reflects a fundamental structural limitation, not just noise.

Failure modes

1. Sparsity collapse (Wide-table DP-VAE)

Flattening 8 tables into one wide table creates extreme sparsity. The VAE encodes most columns as near-zero, collapsing continuous metrics (power, network bytes, RAM) to degenerate distributions.

2. Independence assumption (Per-table DP-SGD, MST)

Generating each table independently destroys foreign-key relationships. Cross-table joins produce mismatched groups, inflating relative error and reducing Jaccard similarity below thresholds.

3. LLM knowledge ceiling (Private Evolution)

The LLM does not know all device vendors, OS versions, or geographic regions in the real dataset. This causes 0% group coverage on queries involving rare categories.

4. DP noise masking signal (all methods)

At ε = 4.0, privacy noise is large enough to mask subtle correlations (e.g., CPU frequency vs. power consumption) and rare subgroups.

What changed after early attempts

  • Initial approach: we started with only the wide-table DP-VAE. Pass rate was 1/8, prompting us to explore per-table splitting.
  • Adding per-table: splitting improved single-table queries dramatically but revealed the join problem.
  • Adding MST: the histogram baseline matched per-table DP-SGD on pass rate (6/21) but with different query strengths.
  • Adding Private Evolution: the training-free method showed promise on targeted workloads but was limited by LLM knowledge.
  • Takeaway: no single method dominates. A hybrid routing strategy (different methods for different query types) is the most promising path forward.

Discussion

Key takeaways

1. Per-table independence breaks cross-table queries

Per-table DP-SGD and MST generated high-fidelity data within individual tables but failed on queries joining multiple tables. Foreign-key mismatches caused inflated relative errors and low Jaccard overlap.

Lesson: protecting per-table privacy is not sufficient for relational workloads. Future methods must reason about foreign-key relationships directly.

2. Wide-table sparsity collapses continuous metrics

Flattening all tables into one wide table introduced extreme sparsity. The VAE encoded most columns as near-zero, destroying continuous distributions (power consumption, network throughput, RAM usage).

3. Training-free methods have a knowledge ceiling

Private Evolution relied on an LLM’s pre-trained knowledge. When real data contained categories unseen by the model (geographic regions, rare OS versions, niche applications), the method produced 0% group coverage on those queries.

4. Simple distributions are tractable

Queries on categorical distributions with few categories (browser usage, vendor percentages) passed across multiple methods. This suggests distribution synthesis is largely solved for non-sparse data.

5. DP noise is necessary but limiting

Even at ε = 4.0 (moderate privacy), noise masked subtle correlations and rare subgroups. Query-specific privacy budgets via advanced composition may help allocate noise more efficiently.

Limitations

  • Single privacy budget: we only evaluated at ε = 4.0. Performance at stricter budgets (ε < 1) may differ substantially.
  • One dataset: results are specific to the Intel DCA corpus. Other domains (healthcare, finance) may show different method rankings.
  • No relational DP: we did not implement methods that reason about foreign keys during synthesis (e.g., DP joins, relational GANs).
  • LLM not fine-tuned: Private Evolution used a general-purpose LLM. Domain-specific fine-tuning could improve category coverage.
  • Benchmark design: our 21 queries, while diverse, may not cover all production workload patterns. Results should be interpreted as indicative, not exhaustive.

Recommendations for practitioners

No single method dominates. We recommend routing queries to different methods based on type:

  • Simple aggregates and distributions: MST (fast, interpretable, good TV scores).
  • Row-level analytics: per-table DP-SGD (best within-table fidelity).
  • Ranking queries: MST (good Jaccard overlap on top-k items).
  • Cross-table joins: requires future relational DP methods; current approaches are insufficient.

Future directions

Relational differential privacy

Develop methods that reason about foreign-key relationships directly, applying DP at the join level. This is the most impactful open problem identified by our work.

Domain-adapted LLMs for PE

Fine-tune LLMs on domain-specific schemas and category lists to improve Private Evolution’s coverage of rare values.

Multi-epsilon evaluation

Characterize the privacy-utility frontier by sweeping ε from 0.1 to 10 for each method.

Broader datasets

Validate findings on healthcare, financial, and census datasets to test generalizability.

Conclusion

Training-based and training-free DP approaches each have strengths and weaknesses. There is no silver bullet. Our evaluation shows that:

  • Simple distributions are tractable; complex joins are not (yet).
  • Per-table and wide-table synthesis represent opposite trade-offs.
  • Hybrid query-aware routing significantly improves overall performance.
  • Future work in relational DP and workload-aware composition is critical.
